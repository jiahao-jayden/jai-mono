# What is Capsule

Capsule lets AI agents output rich UI.

Traditional agents can only reply with text and Markdown. Weather, charts, forms — plain text doesn't cut it. Capsule separates UI from data: UI components are pre-built in React, the agent just provides data, and the host application renders it in a sandbox.

The agent doesn't write code, doesn't generate HTML, doesn't touch the network. All it does is pick a component and fill in the data.

## Problems solved

**Agents can only output text.** Markdown tables sort of show structured data, but there's no interactivity, no style control, no way to embed charts.

**Having LLMs write HTML directly is unreliable.** Model-generated HTML is inconsistent, layouts break, every response regenerates the entire UI with no reuse, and security is hard to lock down.

**UI and data get tangled together.** When the agent handles both UI structure and data, the two concerns interleave and errors multiply.

Capsule's division of labor:

- UI components are pre-built, installed locally, reviewed
- The agent picks a component and provides data
- The host renders it in a sandbox

## Key features

**React component model.** Capsule uses React as its only UI framework. Components are registered via `defineCapsule()`, the host provides the React runtime, and the capsule contains only business code.

**Multi-component support.** A single capsule can register multiple components. For example, a `finance` capsule can include a stock chart, portfolio overview, and transaction list — the agent picks whichever fits.

**One-shot rendering.** Data is injected once at startup, the capsule renders once, done. No data updates, no bidirectional communication, no dispose lifecycle. To show new data, the agent calls the tool again and the host creates a new instance.

**Sandbox isolation.** Capsules run in isolated iframe sandboxes with CSP blocking external network requests. Capsules cannot access the host DOM, make network requests, or read localStorage. All data must be fetched by the agent beforehand and passed in as initial data.

## Design reference

The approach borrows from Google's [A2UI protocol](https://a2ui.org/). A2UI has the client maintain a component registry — the agent just says "use the Text component, content is this."

Capsule differs from A2UI in extensibility. A2UI limits agents to predefined primitives (Text, Card, List) — no custom weather cards or candlestick charts. Capsule allows arbitrary custom UI components — as long as they're built with React and registered via `defineCapsule()`.

The shared principle: **the agent sends data, not code. The client decides how to render.**

## Use cases

Good fit:

- Data display. Weather, stocks, charts — the agent fetches data and passes it to a capsule, much richer than Markdown
- Components needing specific visual design. Brand cards, product showcases — Markdown can't do these
- Standardized information with variable data. Calendar cards, contact cards, order status
- A group of related UIs. One capsule registers multiple components, sharing styles and dependencies

Not a good fit:

- When plain text is sufficient. Don't use a capsule just for aesthetics
- Complex forms with multi-step interaction. Capsules render once, no data updates
- When the UI structure is completely different every time. Capsule UI is predefined, the agent only fills in data
