# Protocol spec

The Capsule Protocol defines four things: how a capsule describes itself (manifest), how code is packaged (bundle), how components are registered (defineCapsule), and how they communicate at runtime (wire messages).

The protocol is host-agnostic. iframes, WebViews, separate processes — all valid.

## Manifest

Every capsule has a `capsule.json` file:

```ts
interface CapsuleManifest {
  protocol: "capsule/v0";
  id: string;
  version: string;
  entry: string;
  components: Record<string, {
    title?: string;
    description?: string;
    dataSchema: JSONSchema;
  }>;
  _meta?: Record<string, unknown>;
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `protocol` | Yes | Fixed value `"capsule/v0"` |
| `id` | Yes | Unique identifier, non-empty string |
| `version` | Yes | SemVer recommended |
| `entry` | Yes | ESM bundle path, relative to the manifest directory |
| `components` | Yes | Component registry, keys are component names |
| `_meta` | No | Extension field, not parsed by the protocol |

### components

Each component has its own `dataSchema` and optional `title` and `description`.

Single-component capsules use `"default"` as the key:

```json
{
  "components": {
    "default": {
      "title": "Weather",
      "description": "Show current weather for a city",
      "dataSchema": { "type": "object", "properties": { "city": { "type": "string" } } }
    }
  }
}
```

Multi-component capsules use meaningful keys:

```json
{
  "components": {
    "stock-chart": {
      "title": "Stock Chart",
      "dataSchema": { ... }
    },
    "portfolio": {
      "title": "Portfolio",
      "dataSchema": { ... }
    }
  }
}
```

`description` gets written into the AI tool description. Be specific about when to use this component so the LLM can judge correctly.

### Forward compatibility

Validators must ignore unknown fields in the manifest rather than rejecting them. Contents of `_meta` are treated as opaque.

## Bundle

`entry` points to a self-contained ESM file.

Requirements:

- All business dependencies bundled in — the sandbox has no module loader
- **React is not bundled.** The host provides the React runtime. Mark `react`, `react-dom`, and `react-dom/client` as external when bundling
- Must have a default export — the return value of `defineCapsule()`
- Must be loadable via dynamic `import()`

```bash
esbuild src/index.tsx --bundle --format=esm --outfile=index.js \
  --external:react --external:react-dom --external:react-dom/client
```

## Component registration

Capsules register components via `defineCapsule()`. This function does not execute rendering — it only returns a registry object.

```ts
import { defineCapsule } from "@jayden/capsule-protocol/runtime";
import { z } from "zod";

export default defineCapsule({
  id: "weather",
  version: "0.1.0",
  components: {
    default: {
      dataSchema: z.object({ city: z.string() }),
      render({ data, theme }) {
        return <div>{data.city}</div>;
      },
    },
  },
});
```

### defineCapsule signature

```ts
function defineCapsule(config: {
  id: string;
  version: string;
  components: Record<string, {
    title?: string;
    description?: string;
    dataSchema: z.ZodType;
    render: (props: CapsuleProps) => ReactElement | null;
  }>;
}): CapsuleRegistry;
```

### CapsuleProps

```ts
interface CapsuleProps<D = unknown> {
  data: D;
  theme: "light" | "dark" | null;
  instanceId: string;
}
```

- `data` — Data from the agent, typed by `dataSchema`
- `theme` — Host's current theme. `null` means the host didn't provide one
- `instanceId` — Instance identifier

### render function

- Returns a React element
- Pure function, should not have side effects
- Does not call `ReactDOM.render()` — the host handles mounting

### How the host uses the registry

After loading the bundle, the host:

1. Gets the default export (`CapsuleRegistry`) via dynamic import
2. Looks up the target component in `registry.components` by key
3. Calls `component.render({ data, theme, instanceId })` to get a React element
4. Mounts with `createRoot(element).render(...)`

Capsule code does not need to be aware of the host's loading mechanism.

## Wire messages

Runtime communication consists of three message types, all from sandbox to host. Transport is unspecified (postMessage, IPC, etc.) — the protocol only defines message shapes.

### capsule/ready

```ts
{ type: "capsule/ready", instanceId: string }
```

Bundle execution complete, first paint done. Optional — hosts should work fine without it.

### capsule/error

```ts
{ type: "capsule/error", instanceId: string, message: string, stack?: string }
```

Uncaught exception or unhandled promise rejection.

### capsule/resize

```ts
{ type: "capsule/resize", instanceId: string, width?: number, height?: number }
```

Root element dimensions changed. The host can use this to adjust container size. Optional.

### General rules

- Every message carries `instanceId`, which must match the host-assigned value
- Hosts should drop messages with unknown `instanceId`
- Hosts should ignore messages with unknown `type` (forward compatibility)
- No host → sandbox messages exist

## Security guidance (non-normative)

The protocol recommends hosts:

- Load capsules in a null-origin sandbox (e.g. `<iframe sandbox="allow-scripts">`)
- Apply strict Content Security Policy to the sandbox
- Validate `data` against `dataSchema` before sending it to the capsule
- Isolate capsule instances — no shared globals

Capsules should not assume network or storage access. External data should be fetched by the agent beforehand and passed via `data`.

## Rendering model

Capsules are one-shot.

Data is injected once at startup, the capsule renders once, done. No data update mechanism, no bidirectional communication, no dispose lifecycle.

To show new data, the agent calls the tool again and the host creates a new capsule instance.

## Versioning

Current protocol identifier: `capsule/v0`. Unstable, breaking changes possible.

`capsule/v1` (future) commits to backward compatibility within the major version:

- Adding fields is non-breaking
- Removing or repurposing fields is breaking — bumps the major version
- Hosts must ignore unrecognized fields and message types
