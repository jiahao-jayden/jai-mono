# Core concepts

## Signal

After the agent calls `RenderCapsule`, the tool result contains an XML tag:

```xml
<jai-capsule id="weather" schema-hash="a3f8c1e2" data='{"city":"Beijing","temp":22,"condition":"Sunny"}' />
```

Three attributes:

| Attribute | Description |
|-----------|-------------|
| `id` | Capsule identifier, matches the install directory name |
| `schema-hash` | First 8 hex chars of SHA-256 of `dataSchema`, used for compatibility checks |
| `data` | JSON object — the data the agent filled in |

The signal is self-contained. The host knows what to render and what data to pass without making additional network requests.

The XML format is a deliberate choice — LLMs generate XML tags more reliably than structured JSON in tool results.

## Catalog

On startup, the host calls `GET /api/plugins/capsule-plugin/catalog` and gets back all capsule manifests and bundle code in one shot, building a local registry.

```json
{
  "capsules": [
    {
      "id": "weather",
      "schemaHash": "a3f8c1e2",
      "manifest": { ... },
      "bundleCode": "const boot = window.__CAPSULE_BOOT__; ..."
    }
  ]
}
```

At render time, lookup is by `id` from memory. No network requests.

This is the A2UI catalog concept in practice — the client holds the full component library, the agent just specifies a name.

## Manifest

Each capsule directory contains a `capsule.json`:

```json
{
  "protocol": "capsule/v0",
  "id": "weather",
  "version": "0.1.0",
  "title": "Weather",
  "description": "Show current weather for a city.",
  "entry": "./index.js",
  "dataSchema": {
    "type": "object",
    "required": ["city"],
    "properties": {
      "city": { "type": "string" },
      "temp": { "type": "number" },
      "condition": { "type": "string" }
    }
  }
}
```

Field reference:

| Field | Required | Description |
|-------|----------|-------------|
| `protocol` | Yes | Must be `"capsule/v0"` |
| `id` | Yes | Unique identifier, must match directory name |
| `version` | Yes | SemVer recommended |
| `entry` | Yes | ESM bundle path, relative to manifest file |
| `dataSchema` | Yes | JSON Schema draft-07 describing the data the agent needs to provide |
| `title` | No | Included in tool description to help the LLM understand what this capsule does |
| `description` | No | Same, more detailed |
| `fallback` | No | `{ "text": "{city}: {temp}°C" }` template for text-only fallback |
| `_meta` | No | Extension field, not parsed by the protocol |

`dataSchema` serves two purposes: the plugin uses it for runtime validation (bad data from the agent returns an error for retry), and it's summarized in the tool description to help the LLM understand what fields to fill.

## Bundle

`entry` points to a self-contained ESM file. React, Vue, Svelte, vanilla JS — all fine. All dependencies must be bundled in. The sandbox has no npm and no network.

When the bundle executes in the sandbox, it reads host-injected startup info from a global:

```js
const boot = window.__CAPSULE_BOOT__;

boot.element      // HTMLElement — mount point
boot.instanceId   // string — instance ID
boot.initialData  // object — data from the agent
boot.theme        // "light" | "dark" | null
```

A minimal capsule is just a few lines:

```js
const boot = window.__CAPSULE_BOOT__;
if (!boot) throw new Error("no-bootstrap");

const el = document.createElement("div");
el.textContent = boot.initialData.message;
boot.element.appendChild(el);
```

`__CAPSULE_BOOT__` only exists in the sandbox environment. If your capsule uses React and exports via `defineCapsule()`, it can be imported as a regular component outside the sandbox (tests, Storybook).

## One-shot rendering

Capsules render once and they're done.

Agent provides data → capsule reads data → renders DOM → finished.

No `onUpdate`. No `onDispose`. No `postAction`. The host never sends messages back to the sandbox.

If the agent needs to show new data, it calls `RenderCapsule` again, generates a new signal, and the host creates a new sandbox instance. The old capsule instance stays in chat history untouched.

This constraint is deliberate. Bidirectional communication and state management would dramatically increase protocol and implementation complexity, and the vast majority of agent UI needs boil down to "got data, show it."
