# Jai 的实现

这一节描述 Jai Desktop 作为第一个宿主，对 Capsule Protocol 的具体实现。这是参考设计，不是协议本身——其他宿主可以有完全不同的做法。

## 三层架构

Jai 把 capsule 拆成三层：

**capsule-plugin**（plugin 层）跑在 gateway 进程里。负责发现本地 capsule、注册 `RenderCapsule` AI 工具、校验数据、生成 signal。它是一个标准的 Jai 插件，安装在 `~/.jai/plugins/capsule-plugin/`。

**单个 capsule**（capsule 层）是 `~/.jai/capsules/<id>/` 下面的一个目录——`capsule.json` 加 `index.js`。Capsule 作者不需要知道 plugin 系统。

**Jai Desktop**（宿主层）解析 signal、从 catalog 查找 capsule、提供 React runtime、在 iframe 沙箱中渲染。

层与层之间的依赖是单向的。Plugin 扫描 capsule，desktop 消费 plugin 的输出。

## Capsule 发现

Gateway 启动时，capsule-plugin 的 `boot()` 扫描 `~/.jai/capsules/`：

- 遍历每个子目录，读取 `capsule.json`
- 校验 manifest 格式和必填字段
- 确认 `manifest.id` 和目录名一致
- 确认 `entry` 指向的文件存在
- 支持符号链接——开发时 `ln -s <repo>/my-capsule ~/.jai/capsules/my-capsule`

结果缓存在内存。Session 级别的工具注册复用 boot 时的扫描结果。

## Signal 格式

Agent 调用 `RenderCapsule` 后，capsule-plugin 返回一段 XML 作为 tool result：

```xml
<!-- 单组件 capsule -->
<jai-capsule id="weather" schema-hash="a3f8c1e2" data='{"city":"Beijing","temp":22}' />

<!-- 多组件 capsule -->
<jai-capsule id="finance" component="stock-chart" schema-hash="b7d2e9f1" data='{"symbol":"AAPL"}' />
```

| 属性 | 说明 |
|------|------|
| `id` | capsule 标识 |
| `component` | 组件名。单组件 capsule 省略，desktop 默认用 `"default"` |
| `schema-hash` | 该 component 的 `dataSchema` 的 SHA-256 前 8 位 |
| `data` | JSON 对象，HTML 实体编码 |

XML 格式是刻意的——LLM 对 XML 标签的生成比结构化 JSON 更稳定。`data` 用单引号包裹，内部双引号编码为 `&quot;`。

Signal 是 Jai 的实现细节，不是协议的一部分。

## Catalog 与按需加载

capsule-plugin 注册两个端点：

### 获取 manifest（启动时调用一次）

```
GET /api/plugins/capsule-plugin/catalog
```

```json
{
  "capsules": [
    {
      "id": "weather",
      "manifest": { "protocol": "capsule/v0", "id": "weather", ... },
      "schemaHashes": { "default": "a3f8c1e2" }
    },
    {
      "id": "finance",
      "manifest": { ... },
      "schemaHashes": { "stock-chart": "b7d2e9f1", "portfolio": "c4a6d8e3" }
    }
  ]
}
```

只返回 manifest 元数据，不含 bundle 代码。

### 获取 bundle（首次渲染时按需拉取）

```
GET /api/plugins/capsule-plugin/capsules/:id/bundle
```

返回该 capsule 的 JS 代码字符串。Desktop 拉取后缓存在内存，后续渲染直接使用。

20 个 capsule 安装着，启动时只传 20 份 JSON（几 KB）。用户触发了 weather，才去拉 weather 的 JS。

## React 注入

Jai Desktop 在 sandbox iframe 中预注入 React runtime：

```html
<script>
  window.React = /* 预打包的 React UMD */;
  window.ReactDOM = /* 预打包的 ReactDOM UMD */;
</script>
```

Capsule 打包时把 React 标记为 external，打包工具将 `import React from "react"` 转为读取 `window.React`。

这意味着：
- Capsule bundle 只包含业务代码，通常几 KB
- 所有 capsule 共享同一份 React，不重复加载
- React 版本由 Jai Desktop 控制

## 沙箱渲染流程

Desktop 用 `<iframe sandbox="allow-scripts" srcDoc="...">` 渲染 capsule。完整流程：

1. 生成 srcDoc HTML
2. `<head>` 注入 CSP：`default-src 'none'; script-src 'unsafe-inline' blob:; style-src 'unsafe-inline';`
3. `<script>` 注入 React/ReactDOM 到 window
4. 将 bundleCode 包装为 Blob URL
5. 执行 host 脚本：

```js
const blob = new Blob([bundleCode], { type: "application/javascript" });
const url = URL.createObjectURL(blob);
const mod = await import(url);
const capsule = mod.default;
const comp = capsule.components[componentKey];
const root = ReactDOM.createRoot(element);
root.render(React.createElement(comp.render, { data, theme, instanceId }));
```

6. ResizeObserver 监测根元素尺寸变化，发 `capsule/resize`
7. `window.onerror` 和 `window.onunhandledrejection` 捕获异常，发 `capsule/error`
8. 渲染完成后发 `capsule/ready`

所有消息通过 `window.parent.postMessage()` 发送。Host 通过 `e.source === iframe.contentWindow` 校验来源。

## Schema 兼容性检测

Signal 里的 `schema-hash` 是 capsule 被调用那一刻该 component 的 `dataSchema` 指纹。

Desktop 渲染逻辑：

1. 按 `id` 从 catalog 找到 capsule
2. 按 `component` 找到对应的 schemaHash
3. 对比 signal 的 `schema-hash` 和 catalog 的 hash
4. 匹配 → 正常渲染
5. 不匹配 → capsule 的 schema 更新了，旧数据可能不兼容，显示错误

效果：

- Capsule 只改了 UI（修 bug、换样式）→ hash 没变 → 旧聊天记录照常渲染
- Capsule 改了数据结构（加字段、改类型）→ hash 变了 → 旧聊天记录显示加载失败

## 数据校验

capsule-plugin 在 `RenderCapsule` 的 `execute` 里做基础校验：

- 检查 required 字段是否存在
- 检查字段类型是否匹配
- 处理 LLM 把 data 传成字符串的情况（自动 JSON.parse）

校验失败返回 `isError: true` 和 schema 摘要，Agent 可以修正后重试。

不是完整的 JSON Schema 校验——不做 pattern、minLength、enum，但能挡住最常见的 LLM 错误。

## RenderCapsule 工具

capsule-plugin 注册一个 `RenderCapsule` 工具，描述中列出所有可用 capsule 和 component。

单组件 capsule 展示为 `id`，多组件展示为 `id/component`：

```
Available capsules:

- weather — 天气
  description: 展示城市当前天气
  data: { city: string (required), temp: number, condition: string }

- finance/stock-chart — K线图
  description: 展示股票价格走势
  data: { symbol: string (required), prices: Array<...> (required) }

- finance/portfolio — 持仓概览
  data: { holdings: Array<...> (required) }
```

工具参数中 `capsuleId` 用 `"weather"` 或 `"finance/stock-chart"` 格式。插件内部解析：不含 `/` 就默认 component 是 `"default"`。
