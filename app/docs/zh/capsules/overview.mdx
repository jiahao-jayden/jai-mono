# Capsule 是什么

Capsule 让 AI Agent 可以输出富 UI。

传统 Agent 只能回复文本和 Markdown。天气、图表、表单这些内容用纯文本表达不好。Capsule 的做法是把 UI 和数据分开——UI 组件预先写好安装在本地，Agent 只负责提供数据，宿主应用把数据注入沙箱渲染。

Agent 不写代码，不生成 HTML，不碰网络。它做的事情就像在表单里填字段。

## 解决的问题

**Agent 只能输出文本。** Markdown 表格勉强能展示结构化数据，但没有交互，没法控制样式，更不能嵌入图表。

**让 LLM 直接写 HTML 不可靠。** 模型生成的 HTML 质量参差不齐，布局容易崩，每次都重新生成整个 UI 没法复用，安全上也很难限制。

**组件和数据耦合。** Agent 既管 UI 结构又管填数据，两件事缠在一起，出错概率翻倍。

Capsule 的分工：

- UI 组件预先写好，安装在本地，经过审查
- Agent 选组件、传数据
- 宿主在沙箱里渲染

## 设计参考

思路借鉴了 Google 的 [A2UI 协议](https://a2ui.org/)。A2UI 让客户端维护组件注册表，Agent 只说"用 Text 组件，内容是这个"。

Capsule 和 A2UI 的区别在扩展性。A2UI 把 Agent 限制在预定义的基础组件里（Text、Card、List），做不了自定义天气卡片或 K 线图。Capsule 允许任意自定义 UI，代价是每个 capsule 要打包成独立 JS bundle。

两者的核心原则一样：**Agent 发数据，不发代码。客户端决定怎么渲染。**

## 适用场景

适合：

- 数据展示。天气、股票、图表——Agent 拿到数据传给 capsule，比 Markdown 效果好得多
- 需要特定视觉设计的组件。品牌卡片、产品展示，Markdown 做不到
- 格式固定、数据可变的信息。日程卡片、联系人卡片、订单状态

不适合：

- 纯文本够用的场景。不需要为好看而强行用 capsule
- 多轮交互的复杂表单。Capsule 是一次性渲染，不支持数据更新
- 每次 UI 结构都不同。Capsule 的 UI 预定义，Agent 只填数据
