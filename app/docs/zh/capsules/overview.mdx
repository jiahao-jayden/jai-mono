# Capsule 是什么

Capsule 是 Jai 中 AI Agent 输出富 UI 的机制。

传统的 Agent 只能输出文本和 Markdown。遇到天气、图表、表单这类内容时，纯文本的表达力不够。Capsule 让 Agent 可以调用一个工具，输出一段结构化的标记，宿主应用读到后渲染成真正的交互式 UI 组件。

## 解决的问题

**Agent 输出只有文本。** Markdown 表格勉强能展示结构化数据，但没法做交互，没法控制样式，更没法嵌入图表或自定义的可视化。

**直接让 LLM 写 HTML 不可靠。** 模型生成的 HTML 质量参差不齐，布局容易崩。每次都重新生成整个 UI，没法复用。安全上也麻烦——你不想让 Agent 随意发起网络请求或操作 DOM。

**组件和数据耦合在一起。** 如果 Agent 既要负责 UI 结构又要负责填数据，两件事缠在一起，出错的概率翻倍。

Capsule 把这两件事分开了：

- UI 组件是预先写好的，安装在本地，经过审查
- Agent 只负责"选哪个组件"和"传什么数据"
- 宿主把数据注入到沙箱化的组件里渲染

Agent 不写代码，不生成 HTML，不碰网络。它做的事情就跟在表单里填字段差不多。

## 设计参考

这个思路借鉴了 Google 的 [A2UI 协议](https://a2ui.org/)。A2UI 的做法是客户端维护一个组件注册表（catalog），Agent 只说"用 Text 组件，内容是这个"。客户端自己知道怎么渲染 Text。

Capsule 和 A2UI 的区别在扩展性上。A2UI 把 Agent 限制在一组预定义的基础组件里（Text、Card、List 这些），你没法做自定义的天气卡片或者 K 线图。Capsule 允许任意自定义 UI——代价是每个 capsule 需要打包成独立的 JS bundle。

两者共同的核心原则是一样的：**Agent 不发代码，只发数据。客户端自己决定怎么渲染。**

## 适用场景

适合用 Capsule 的情况：

- 数据展示类的 UI。天气、股票、数据图表。Agent 拿到数据后直接传给 capsule，比用 Markdown 呈现效果好得多。
- 需要特定视觉设计的组件。品牌卡片、产品展示。这些靠 Markdown 做不到。
- 标准化的信息展示。日程卡片、联系人卡片、订单状态。格式固定、数据可变的场景天然适合 capsule。

不适合的情况：

- 纯文本回答足够的场景。不需要为了好看而强行使用 capsule。
- 需要多轮交互的复杂表单。Capsule 是一次性渲染的，不支持数据更新和双向通信。
- 每次 UI 结构都完全不同的场景。Capsule 的 UI 是预定义的，Agent 只填数据。如果每次都需要不同的布局，capsule 帮不了你。

## 三层架构

Capsule 的实现分三层，各自独立：

**Plugin 层。** `capsule-plugin` 是一个 Jai 插件，负责发现本地安装的 capsule、注册 `RenderCapsule` 工具、校验数据、生成 signal。它跑在 gateway 进程里。

**Capsule 层。** 单个 capsule 就是一个目录：`capsule.json`（manifest）+ `index.js`（bundle）。放在 `~/.jai/capsules/<id>/` 下面。capsule 作者不需要知道 plugin 系统的存在。

**宿主层。** 桌面端或其他宿主应用，负责解析 signal、从 catalog 查找 capsule、在沙箱中渲染。Jai Desktop 是第一个宿主实现。

这三层之间的依赖是单向的：plugin 扫描 capsule，宿主消费 plugin 输出的 signal。层与层之间通过 manifest 和 signal 格式解耦。
