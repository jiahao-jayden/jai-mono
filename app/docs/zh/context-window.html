探索上下文窗口

编码助手（如 Claude Code）在一次会话里"知道"的一切——你的指令、它读过的文件、它自己说过的话， 甚至一些从不出现在终端里的内容——都装在同一个上下文窗口里。下面这个小模拟按时间顺序演示了 从你敲下启动命令，到上下文写满触发压缩为止，窗口里都发生了什么。播放到你发消息的节点会停下来， 点「发送」才会继续；平时悬停任意一行可以预览，点一下能固定住方便你滚动查看。

说明：本页基于 Claude Code 官方文档"Explore the context window"一文所讲解的概念与部分交互机制 （例如分阶段暂停等待发送、终端可见性分级、子代理独立上下文的嵌套展示），用自己的场景、文字、 数值和代码重新实现，并非官方页面的逐句翻译或源码移植。若需查阅权威原文，请前往 Claude Code 官方文档站点的同名页面。

这条时间线在演示什么

整个模拟按四个阶段展开，token 数字都是示意性的，用来体现相对量级，不代表真实精确值：

1. 会话启动时（你还没打字）

系统提示词、项目里的 CLAUDE.md、自动记忆、已注册的技能描述、以及可用 MCP 工具的名称与说明，会在你输入任何内容之前就全部进入上下文。你自己的配置可能还会追加更多， 比如自定义的输出风格，或者启动时通过参数追加的额外系统提示——它们都会并入同一份系统提示词。

2. Claude 工作过程中

每读一个文件都会往上下文里加内容；如果这个文件路径命中了某条"路径限定规则"，规则也会跟着自动加载； 每次编辑完成后，如果配置了 PostToolUse 钩子，钩子的执行结果也会写回上下文——但钩子本身 是代码执行，不会出现在你的终端里。

3. 追加需求时

当你临时提出一个需要大量翻阅文件的调研请求时，助手可以把这部分工作交给一个子代理， 在它自己独立的上下文窗口里完成——大量文件内容不会挤占你当前对话的空间， 主对话里只会收到一段摘要和一点点元数据。

4. 上下文快写满时

触发 /compact 后，冗长的对话历史会被替换成一份结构化摘要。 大部分启动阶段加载的内容会自动重新注入，但具体机制各不相同——下表列出了每一类内容在压缩后的命运。

压缩之后，哪些内容会留下

长会话触发压缩时，Claude Code 会把对话历史总结成摘要以腾出空间。你的各类指令是否留存，取决于它们最初是怎么加载进来的：

加载方式压缩之后
系统提示词 / 输出风格不受影响，它们本来就不属于对话历史
项目根目录 CLAUDE.md、未加路径限定的规则从磁盘重新注入
自动记忆从磁盘重新注入
带 paths: 限定的规则丢失，直到再次读取匹配的文件才会重新加载
子目录里的嵌套 CLAUDE.md丢失，直到再次读取该子目录下的文件
已调用的技能正文会重新注入，但每个技能上限约 5,000 tokens、全部技能合计上限约 25,000 tokens，超出部分优先丢弃较早调用的技能
钩子（Hooks）不涉及，钩子是代码执行，本身不占用上下文

路径限定的规则和嵌套 CLAUDE.md 是随对话消息一起加载的，所以压缩时会连同其余对话一起被总结掉， 下次读到匹配文件时才会再次出现。如果某条规则必须在压缩后依然生效，去掉它的 paths: 限定，或者把内容挪到项目根目录的 CLAUDE.md 里。技能正文的截断策略是保留文件开头部分， 所以重要说明应该写在 SKILL.md 靠前的位置。

上下文快写满时可以做什么

Claude Code 会在接近上限前自动压缩，所以上下文写满本身不会中断会话；自动压缩的效果和上面 /compact 那一步是一样的。除此之外，你也可以在自动压缩发生之前主动出手：

  • 带着重点手动压缩：开始一段新的长任务前，运行 /compact 聚焦在登录鉴权那个 bug 上 这样带说明的压缩，摘要会保留你指定的重点，而不是让自动压缩去猜。
  • 切换任务就清空：处理不相关的新任务时运行 /clear。留着旧对话不但占地方，还会让接下来每条消息都多花 token。
  • 把大批量阅读甩给子代理：需要大量翻阅文件做调研时，交给子代理去做，文件内容留在它自己的上下文里，不进你的主对话。
  • 换一个更大的窗口：如果问题是"任务本身就需要很大的窗口"而不是"对话太啰嗦"，部分模型支持百万级 token 的扩展上下文，可以按需切换。

查看你自己会话的真实数据

本页的模拟数字只是示意。想知道当前这次会话真实的上下文占用情况，随时运行 /context 查看按类别拆分的实时用量和优化建议；运行 /memory 可以确认启动时到底加载了哪些 CLAUDE.md 和自动记忆文件。
