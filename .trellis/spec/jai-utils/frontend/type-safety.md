# Type Safety

> Type safety patterns in `@jayden/jai-utils`.

---

## Overview

`jai-utils` uses TypeScript with Zod for runtime validation. The package defines two main type families: the `NamedError` class hierarchy and the `ParsedModelId` type.

---

## Type Organization

All types are co-located with their implementation files and re-exported from the barrel `src/index.ts`:

- `ParsedModelId` -- defined in `src/model-id.ts`, exported as `type`
- `NamedError` -- abstract class in `src/error.ts`, exported as a value (class)

---

## Validation

Zod is the runtime validation library. `NamedError.create()` accepts a Zod schema for the error data, which provides:

1. **Compile-time type inference** via `z.input<Data>`
2. **Runtime schema** accessible via the static `Schema` property
3. **Serialization** via `toObject()` returning `{ name, data }`

Example of the Zod schema pattern:

```ts
// The schema is a ZodObject with a literal name and the data schema
const MyError = NamedError.create(
  "MyError",
  z.object({ code: z.number(), detail: z.string() }),
);

// MyError.Schema = z.object({ name: z.literal("MyError"), data: z.object({ code, detail }) })
// new MyError({ code: 404, detail: "..." }) -- data is type-checked
```

---

## Common Patterns

### Type guard via `isInstance()`

Each `NamedError` subclass generated by `create()` has a static `isInstance()` type guard:

```ts
const ModelNotFoundError = NamedError.create("ModelNotFoundError", z.string());

if (ModelNotFoundError.isInstance(err)) {
  // err is narrowed to NamedError & { name: "ModelNotFoundError", data: string }
}
```

### Optional return types

Functions return `undefined` for invalid input rather than throwing. Callers must handle the `undefined` case:

```ts
const result = parseModelId("invalid-no-slash");
// result is undefined
```

---

## Forbidden Patterns

- **Do not use `as` type assertions** to work around `NamedError` types -- use `isInstance()` instead.
- **Do not use `any` for error data schemas** -- always provide a concrete Zod schema to `NamedError.create()`.
- **Do not bypass Zod validation** -- the `NamedError` constructor expects data matching the Zod schema. Passing unvalidated data defeats the purpose.
- **Do not use raw `Error` class** in packages that depend on `jai-utils` -- use `NamedError.create()` for structured, serializable errors.
